The admin table is used to store ADMIN account information including a unique admin ID, a username, and a password. Additional details about the administrator, such as their first and last names, gender, email address, and phone number, are maintained in the admin_info table.

The item-info table is used to store detailed information about medicines and products. This includes the product name, category classification, pricing information, stock quantity available, expiry dates for batch tracking, shelf location numbers, and supplier details. The system automatically updates stock levels when transactions occur in the POS module and generates alerts when stock falls below reorder thresholds.

Database Schema

Tables are the primary structure used to organize the data in a relational database. Each table denotes a particular kind of data, and each column denotes a particular quality of that data. This facilitates data organization, search, and retrieval, enabling users to locate the information they require in a timely manner.

Database Relationships

The database follows a normalized structure with the following key relationships:

  • One-to-Many (1:N): One customer has many orders; one sale has many sale_items; one product appears in many transactions
  • Foreign Keys: Ensure referential integrity and maintain data consistency across related tables
  • Primary Keys: Uniquely identify each record in a table
  • Indexes: Applied to frequently queried columns (product_id, customer_id, sale_id) for optimized performance
  • Constraints: NOT NULL, UNIQUE, CHECK constraints enforced at the database level for data quality
Data Integrity & Normalization

The database schema follows relational database normalization principles:

  • First Normal Form (1NF): All tables contain atomic values with no repeating groups
  • Second Normal Form (2NF): All non-key attributes are fully dependent on the primary key
  • Third Normal Form (3NF): No transitive dependencies between non-key attributes
  • Referential Integrity: Foreign keys maintain relationships between tables without orphaned records
  • Data Validation: Constraints prevent invalid data entry at the database level
